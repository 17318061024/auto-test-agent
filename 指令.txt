领导明确指示要做成一个桌面客户端的自动化测试工具 
 1.用户一行命令 实现安装在本地电脑 
 2.网页端一键传给客户端指令 客户端开始从服务端读取自动化测试脚本并执行任务,并且将结果传递给服务器 
 3. 客户端提供日志查看的能力 查看哪一个出错了 哪一步耗时多久


项目初始化：创建 Electron + Node.js 项目，配置全局 CLI 命令 midscene-agent。
环境自检模块 (Self-Checker)：
编写一个 EnvironmentMonitor 类，启动时自动检查：Node.js 版本、Chrome 安装路径、端口 9222 占用情况、剩余内存及磁盘空间。
自动修复：如果缺少 Playwright 浏览器二进制文件，自动执行 npx playwright install chromium。
CLI 实现：midscene-agent setup 用于注册 midscene:// 自定义协议并配置开机自启。


执行器封装：基于 @midscene/web 封装 TaskRunner。
自动优化 (Auto-Optimization)：
在执行 从服务端读取自动化测试脚本前，自动检测页面加载状态（Network Idle）。
智能重试机制：若 aiAction 失败，引擎需自动重试（最多3次），并在重试前尝试“等待 2 秒”或“刷新页面”作为补偿策略。
影子执行模式：支持 headful 模式让用户看到 UI，同时后台静默记录所有 DOM 变动。
上下文保持：确保跨页面的任务（如：登录后跳转）能自动传递 Page 对象和 Session。


性能埋点：使用 performance.now() 对以下环节进行测速：
浏览器拉起耗时
页面加载耗时
视觉渲染耗时
执行追踪 (Tracing)：
记录每一步的 step_id、status、duration。
自动诊断问题：如果某一步耗时超过阈值（如 10s），自动标记为“性能瓶颈”，并在日志中高亮展示。
视觉证据库：每一步操作后，自动生成一张包含“红框标记元素”的截图，并与 AI 指令关联存储。



实时看板：
使用步骤条展示执行流。每个步骤点击可展开，显示“耗时详情”。
错误诊断助手：
当任务失败时，UI 需自动展示“失败原因分析”（如：元素不存在、网络超时、断言失败）。
自动优化建议：若因 AI指令 没找准元素失败，界面提供一个“纠错”按钮，允许用户手动指定元素后重新运行。
本地离线日志：使用 SQLite 或 JSON 文件存储历史记录，支持按时间、状态、任务 ID 搜索。


协议拦截器：当通过 midscene://run?taskId=xxx 唤起时，Electron 需自动前置并显示任务执行进度条。
自动化回传：
任务结束后，自动生成包含所有截图、耗时数据、报错信息的 JSON 报文。
实现 流式上传：每完成一步，实时将状态同步给云端 WebSocket，让网页端也能实时看到“执行到第几步”。
最终报告生成：利用 midscene.dumpReport() 导出 HTML 报告，并提供一个“一键发送至邮箱/群机器人”的功能。


编写元测试：创建一个测试脚本。模拟网页下发一个包含“故意出错”步骤的任务，检查 Agent 是否能正确捕获错误、记录耗时并回传给服务器。
鲁棒性检查：模拟网络波动、Chrome 被手动关闭等异常场景，确保 Agent 不会崩溃且能给出明确提示。


补充一个mock示例,   1.点击浏览器页面上的satrt按钮,  自动启动agent, 自动读取服务端的脚本指令  例如（1.打开页面https://www.google.com/webhp  2.输入框中输入  华为  3. 点击按钮AI模式） 将结果返回给服务端 并打印在控制台上
